# Token 管理设计

日期：2026-03-18  
状态：聊天中已确认，工作区已实现  
范围：`douyin` 与 `tiktok` 的首版下载凭证管理

## 1. 目标

在 `设置` 页面增加独立的 Token 管理区域，使下载请求可以使用稳定的平台 Cookie，而不是依赖匿名请求。

首版范围刻意收敛，只包含：

- 手动粘贴
- 保存 / 清空
- 校验
- 状态展示
- 基础获取指引

这一版不尝试浏览器自动导入，也不做内嵌登录。

## 2. 产品定位

Token 管理属于 `设置`，不应分散到各个下载页面里。

原因：

- 凭证状态是全局的
- 多个下载页面会复用同一平台 Cookie
- 校验与过期提示需要集中管理

下载页面可以提示风险，但配置入口只放在 `设置`。

## 3. 范围

支持平台：

- `douyin`
- `tiktok`

每个平台独立维护以下状态：

- cookie 值
- 最近更新时间
- 最近校验时间
- 最近校验状态
- 最近校验消息

## 4. 交互结构

`设置` 页新增 `Token 管理` 分组，包含三部分：

1. 总览卡片
2. 抖音 Token 卡片
3. TikTok Token 卡片

每个平台卡片包含：

- 完整 Cookie 文本框
- `保存`
- `清空`
- `校验`
- `打开官网`
- `获取指引`
- 更新时间 / 校验时间 / 结果等元信息

## 5. 状态模型

固定状态值：

- `not_configured`
- `unchecked`
- `checking`
- `valid`
- `invalid`
- `expired`

含义：

- `not_configured`：未保存 Cookie
- `unchecked`：Cookie 已变化或尚未校验
- `checking`：正在校验中
- `valid`：当前校验通过
- `invalid`：格式错误或校验明确失败
- `expired`：Cookie 疑似过期，需要刷新或重新登录

## 6. 用户流程

### 6.1 保存流程

1. 用户粘贴 Cookie
2. 系统标准化空白与换行
3. 按平台保存 Cookie
4. 更新 `updatedAt`
5. 校验状态重置为 `unchecked`

### 6.2 校验流程

1. 用户点击 `校验`
2. 如果编辑器内容与已保存内容不同，先保存再校验
3. 后端执行平台专属校验
4. 持久化校验结果
5. UI 刷新状态与提示消息

### 6.3 下载页提示流程

下载页不阻止提交，但在以下情况下给出提示：

- 未配置 Cookie
- Cookie 状态为 `unchecked`
- Cookie 状态为 `invalid` 或 `expired`

这样既保持首版可用性，又能引导用户获得更稳定的下载结果。

## 7. 校验策略

校验由 Python 负责，因为下载运行时已经依赖 `f2`。

### 7.1 Douyin

使用 `f2` 的相关能力，基于提供的 Cookie 发起轻量鉴权请求。

结果解释：

- 成功 => `valid`
- 明确登录失效 / 过期信号 => `expired`
- Cookie 格式错误或明显无效 => `invalid`
- 平台抖动 / 网络异常 => `unchecked`

### 7.2 TikTok

使用较轻的启发式校验：

- 校验关键鉴权 Cookie 字段是否存在
- 携带 Cookie 请求首页
- 检测跳转 / 登录信号

结果解释：

- 已鉴权响应 => `valid`
- 跳转登录页 / 登录提示 => `expired`
- 缺少必要字段 => `invalid`
- 瞬时失败 => `unchecked`

## 8. 持久化

所有 Token 状态都存放在设置元数据中，而不是单独建表。

原因：

- 范围小
- 当前只需要最新生效状态，不需要审计历史
- 首版集成成本更低

## 9. 安全约束

首版规则：

- 不自动从浏览器导入
- 不从第三方服务获取 Token
- 仅存储平台值与校验元数据
- 明确提示不要使用未知来源 Cookie

## 10. 非目标

首版暂不包含：

- 浏览器自动导入
- 浏览器扩展集成
- 内嵌登录流程
- 多账号管理
- 自动刷新提醒
- 提交前强制拦截
