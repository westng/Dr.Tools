# 视频下载设计

日期：2026-03-18  
状态：聊天中已确认草案  
范围：仅规划下载功能，不包含录制或更大范围的平台扩展实现细节

## 1. 目标

通过把 `f2` 集成到现有桌面壳层后面，为 Dr.Tools 引入首个可用版的视频下载能力。

这一版范围刻意收敛：

- 平台：`douyin`、`tiktok`
- 功能：仅视频下载
- UI 范围：一级菜单下的一个下载页面
- 执行模型：后台任务队列

目标是先建立稳定的任务模型和页面结构，后续再扩展到录制、主页作品、合集以及更丰富的下载选项，而不需要重写整体壳层。

## 2. 非目标

首版不包含：

- 作者主页下载
- 合集或播放列表下载
- 收藏下载
- 关键词搜索
- 评论或弹幕抓取
- 作者资料抓取
- 提交前解析预览与确认流程

## 3. 产品范围

当前一级导航保持为：

- 视频下载
- 直播录制
- 任务记录
- 设置

下载功能所在位置：

- `视频下载`

页面采用单页任务流，而不是向导式流程。

## 4. 用户流程

首版采用直接提交流：

1. 用户打开 `视频下载`
2. 用户选择平台：`抖音` 或 `TikTok`
3. 用户粘贴一条或多条视频链接
4. 用户按需勾选附加下载项
5. 用户点击 `开始下载`
6. 前端校验、标准化、移除空行并去重
7. 前端提交本批次
8. 后端将批次拆成多个独立任务
9. 任务进入后台队列，按全局并发限制执行
10. 用户从处理日志和任务记录查看结果

v1 不提供预览确认阶段。

## 5. 输入模型

页面接受：

- 单条视频链接
- 一次粘贴多条视频链接

规则：

- 一行一条链接
- 空行忽略
- 提交前去重
- 不接受作者主页、合集等混合输入

## 6. 页面结构

`视频下载` 页面使用标准任务页结构：

### 6.1 输入区

- 多行文本输入
- 一行一条链接的辅助说明
- 平台选择器：`抖音` / `TikTok`

### 6.2 参数区

- 附加内容选择
- 下载参数选择

### 6.3 执行区

- `开始下载` 主操作
- 提交前校验反馈

### 6.4 日志区

- 单条任务处理日志
- 最新优先排序
- 展示当前批次下每条下载任务的处理状态

## 7. 任务模型

批量输入不作为一个不透明的大任务保存。

而是：

- 一条链接映射一个独立下载任务
- 一次批量提交创建多个子任务
- 批次作为一次提交行为存在，用于归档与查看

这样设计的原因是：

- 任务记录更清晰
- 重试粒度更小
- 失败处理更直接
- 后续增加暂停 / 恢复 / 取消更容易

任务类型：

- `video.download`

每个子任务至少应包含：

- `id`
- `taskType`
- `platform`
- `sourceUrl`
- `status`
- `savePath`
- `errorMessage`
- `createdAt`
- `updatedAt`

## 8. 队列与并发

下载调度与任务创建分离。

- 用户提交一个批次
- 系统为每条链接创建一个任务
- 队列执行器按全局并发限制运行任务

并发限制由设置统一管理。

初始设置项：

- `maxConcurrentDownloads`

该设置属于 `设置`，不应放在下载页内。

## 9. 下载内容

首版下载内容：

- 视频文件
- 基础元数据

可选内容：

- 封面图
- 原声
- 文案
- 歌词

首版不下载：

- 评论
- 弹幕
- 作者资料
- 额外互动数据集

## 10. 设置影响

首版新增一个下载相关设置：

- `maxConcurrentDownloads`

复用已有设置：

- 下载路径

下载页直接读取全局目录配置，不再单独维护页面级输出目录设置。

## 11. 数据流

规划中的端到端流程：

1. Vue 页面收集输入与选项
2. 前端标准化链接
3. Rust 命令接收提交载荷
4. Rust 为每条链接写入一个任务记录到 SQLite
5. Rust 队列层把任务派发给 Python / `f2`
6. Python 执行平台专属下载
7. Rust 更新任务状态并写入日志 / 结果
8. 前端读取任务记录、批次明细和处理日志

## 12. 错误处理

预期的首版错误类型：

- 不支持或格式错误的链接
- 标准化后为空提交
- 队列提交失败
- Python 运行时失败
- 平台下载失败
- 文件系统写入失败

要求：

- 单条无效链接不能拖垮整个批次
- 每个失败项都要能在历史记录中独立看到
- 即使其他项失败，成功项也必须保持成功
